今日内容

在前 11 周的学习中，Rust 编译器像一个严格的守护者，阻止我们做任何"危险"的操作：悬挂指针、数据竞争、未定义行为。这些保障让 Rust 成为系统编程的安全选择。

但有时候，我们需要打破这道边界——直接与操作系统交互、调用 C 库、实现自己的内存管理策略、与硬件对话。这时候，unsafe 就是那把钥匙。

1. unsafe 是什么？

Rust 中的 unsafe 不是"危险"的代名词，而是"编译器无法保证安全"的操作空间。你明确告诉编译器：“这部分逻辑我负责，你不用检查。”

unsafe 能做什么（safe Rust 不能做的 5 件事）：

  1. 解引用 raw pointer（*const T 和 *mut T）
  2. 调用 unsafe 函数或方法
  3. 访问或修改可变静态变量
  4. 实现 unsafe trait
  5. 访问 union 的字段

fn main() {
    // 语法上，unsafe 代码和普通代码完全混用
    // 只需在操作前加 `unsafe { ... }` 块
    unsafe {
        // 解引用 raw pointer
        let ptr = 0x1234 as *const i32;
        println!("地址上的值: {:?}", *ptr);
    }
}

核心原则： unsafe 只是解除了编译器的检查，但程序的内存安全仍然是你自己的责任。

2. Raw Pointer（原始指针）

Rust 有两种 raw pointer：

1
2

// *const T —— 不可变指针，不能修改指向的数据
// *mut T   —— 可变指针，可以修改指向的数据

Raw pointer 的特点：

  • 不保证指向有效内存
  • 不保证不空（除非初始化的）
  • 不保证指向的内存未释放
  • 可以在 safe 代码中创建，但只能在 unsafe 块中解引用

fn main() {
    let x = 42;
    let raw_ptr = &x as *const i32;  // safe: 创建 raw pointer
    let mut_var = 0i32;
    let raw_mut = &mut mut_var as *mut i32;  // safe: 创建可变 raw pointer

    unsafe {
        println!("raw_ptr 指向: {}", *raw_ptr);       // 解引用：必须 unsafe
        println!("raw_mut 可变指向: {}", *raw_mut);   // 解引用：必须 unsafe
        *raw_mut = 100;                               // 修改：必须 unsafe
        println!("修改后: {}", mut_var);
    }
}

为什么需要 raw pointer？

  1. 与 C/Fortran 等语言交互
  2. 实现自己的内存分配器
  3. 绑定底层 API（内核、驱动、MMIO）
  4. 在某些性能敏感场景手动优化内存布局

// 安全函数：编译器保证不会 UB
fn safe_div(a: i32, b: i32) -> i32 {
    a / b  // 如果 b=0 会 panic，但不会 UB
}

// unsafe 函数：调用者必须确保前提条件
unsafe fn dangerous_div(a: i32, b: i32) -> i32 {
    // 如果 b=0，结果是 UB（未定义行为）
    // 调用者必须自己保证 b != 0
    a / b
}

fn main() {
    // safe 代码可以调用 unsafe 函数，但必须满足契约
    unsafe {
        let result = dangerous_div(10, 2);
        println!("结果: {}", result);

        // 如果传 b=0，程序行为未定义——可能 panic，可能返回垃圾值，可能崩溃
        // dangerous_div(10, 0); // 不要这样做！
    }
}

}

4. Safe Wrapper（安全封装）

unsafe 的最佳实践：把 unsafe 代码封装在 safe 抽象里。让使用者的代码仍然是 safe 的。

/// 安全的数组排序——内部用了 unsafe 做内存操作，但对外是安全的
fn sort_integers(arr: &mut [i32]) {
    // 标准库内部用了大量 unsafe，但排序接口是 safe 的
    arr.sort();
}

// 标准库的 Vec::set_len 也是 unsafe——但调用条件被严格文档化
// 调用者必须保证 new_len <= capacity，且元素初始化有效
unsafe impl<T> Vec<T> {
    pub unsafe fn set_len(&mut self, new_len: usize) {
        self.len = new_len;
    }
}

标准库模式： 把不安全的底层操作封装成 unsafe fn，但提供 safe 的高级 API。

// 不可变静态变量：可以安全地在多线程间共享
static GREETING: &str = "Hello, Rust!";

// 可变静态变量：访问需要 unsafe 块，否则编译错误
static mut COUNTER: i32 = 0;

// 安全的封装方式：用线程安全类型包裹
use std::sync::atomic::{AtomicI32, Ordering};
static ATOMIC_COUNTER: AtomicI32 = AtomicI32::new(0);

fn main() {
    println!("GREETING: {}", GREETING);
    println!("ATOMIC_COUNTER: {}", ATOMIC_COUNTER.load(Ordering::SeqCst));

    // 直接修改可变形静态变量（需要 unsafe）
    unsafe {
        COUNTER += 1;
        println!("COUNTER (unsafe): {}", COUNTER);
    }

    // 线程安全的方式（推荐）
    ATOMIC_COUNTER.fetch_add(1, Ordering::SeqCst);
}

为什么可变形静态变量是 unsafe 的？

数据竞争（data race）——两个线程同时修改同一块内存，且至少有一个写操作。这是未定义行为。编译器禁止 safe 代码直接访问可变形静态变量。

        let ptr = malloc(100);
        println!("分配了 100 字节 at {:?}", ptr);

        // 使用完毕后释放
        free(ptr);
        println!("已释放内存");
    }
}

关键概念理解

unsafe 代码的安全边界 

1
2
3
4
5

Safe Rust   →  编译器保证内存安全 + 线程安全
    ↓ unsafe 块
Unsafe Rust →  你自己保证（编译器放行）
    ↓ safe 抽象封装
回到 Safe Rust 使用

核心思路： unsafe 不是用来写"不安全"代码的，而是用来实现"底层细节"，再封装成 safe 接口给使用者。

Raw Pointer vs Reference（&T 和 &mut T）

特性 &T / &mut T *const T / *mut T
借用检查 编译器强制 不强制
线程安全 Send/Sync 自动实现 需要自己保证
空指针检查 编译期检查 不检查
生命周期 编译器追踪 不追踪
解引用 safe 必须 unsafe

extern "C" 与 ABI

  • extern "C"：使用 C 的调用约定（ABI）
  • extern "Rust"：使用 Rust 的调用约定（默认）
  • 其他：extern "sysv64"、extern "win64" 等针对不同平台的调用约定

今日感悟

今天打破了 Rust 的"安全壁垒"，但最大的收获其实是理解了为什么 safe Rust 需要 unsafe。

unsafe 不是为了"为所欲为"，而是为了扩展 Rust 的能力边界。比如：

  • 调用 C 库 —— 世界不是只有 Rust 代码
  • 硬件操作 —— 某些场景必须直接操作内存地址
  • 性能优化 —— 在确定安全的场景跳过边界检查

关键点是：unsafe 代码必须用 safe 代码封装，给使用者提供"安全接口"。这样即使底层是 unsafe 的，调用者依然可以是 safe Rust。

第一次在 Rust 里看到裸的 raw pointer 有点不适应——它就像一个"不受编译器管理的借用"。但当你理解了"编译器的所有权检查是在保护你"之后，就明白 unsafe 只是把"保护者检查"换成了"自己检查"。责任更大了，但自由度也更高了。
