回顾：async fn 是什么？

昨天我们学到 async fn 返回一个 Future，这个 Future 必须被"驱动"才能执行。驱动 Future 的机制就是执行器（Executor）。

1

调用 async fn
      ↓
返回一个 Future（惰性的，不执行）
      ↓
执行器调用 Future.poll()
      ↓
如果返回 Pending，Future 暂停；等就绪后再 poll
      ↓
Future 最终返回 Ready(output)

今天我们就深入到 Future 的底层，理解 poll、Pin、Context 和 Waker 是如何协同工作的。

一、Future trait 详解

标准定义

pub trait Future {
    // Future 最终产生的值的类型
    type Output;

    // poll 是执行器调用的唯一方法
    // 调用时机：当 Future 可以取得进展时
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

pub enum Poll<T> {
    Ready(T),   // Future 完成，产出值
    Pending,    // Future 还没准备好，需要等待
}

关键点：

  • poll 不阻塞：当返回 Pending 时，它立即返回，不会等待
  • 需要被多次调用：Future 可能在多次 poll 调用之间交替 Ready 和 Pending
  • Pin<&mut Self>：Future 被"钉"在内存中，不能移动（因为它可能包含自引用结构）

use std::future::Future;
use std::task::{Context, Poll};
use std::pin::Pin;
use std::time::{Duration, Instant};

// 一个延迟返回值的 Future
struct Delay {
    deadline: Instant,
}

impl Future for Delay {
    type Output = &'static str;  // 完成后返回这个字符串

    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        let now = Instant::now();
        
        if now >= self.deadline {
            // 时间到了，Future 完成
            println!("[Delay] 时间到，返回 Ready");
            Poll::Ready("延迟任务完成！")
        } else {
            // 时间还没到，注册 Waker 以便被唤醒
            let waker = cx.waker().clone();
            let deadline = self.deadline;
            
            // 启动一个线程来在到期时唤醒
            std::thread::spawn(move || {
                let until = deadline.saturating_duration_since(Instant::now());
                std::thread::sleep(until);
                waker.wake();  // 唤醒执行器
            });
            
            println!("[Delay] 时间未到，返回 Pending");
            Poll::Pending
        }
    }
}

fn main() {
    let runtime = tokio::runtime::Builder::new_current_thread()
        .build()
        .unwrap();

    let result = runtime.block_on(async {
        let delay = Delay {
            deadline: Instant::now() + Duration::from_millis(100),
        };
        delay.await
    });

    println!("结果：{}", result);
}

struct MyFuture {
    data: String,
    // 假设 some_async_operation 返回一个 Future
    // 这个 Future 引用了 self.data
}

impl Future for MyFuture {
    type Output = String;
    
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        // 如果 self 可以移动，那么内部的引用就会失效
        // Pin 保证了 self 不会被移动
        let future = self.get_mut().some_future.as_mut().unwrap();
        Pin::new(future).poll(cx)
    }
}

Pin 的核心保证：被 Pin 住的值，其内存布局不会被改变。这意味着 &mut Self 可以安全地生成自引用指针。

三、Context 与 Waker

为什么需要 Waker？

poll 方法返回 Pending 时，执行器会等待。但执行器怎么知道什么时候再次调用 poll？

答案是 Waker。

当 Future 处于 Pending 状态时，它需要告诉执行器"等我有数据了，叫我一声"。这个机制通过 Waker 实现。

use std::task::{Context, Poll, Waker};
use std::fmt::Debug;

// Waker 包含一个指向任务上下文的指针
// 调用 waker.wake() 会通知执行器：该任务可以继续了

// 示例：模拟一个 I/O 完成的 Future
struct IoFuture {
    data: Option<String>,
}

impl Future for IoFuture {
    type Output = String;

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        if let Some(data) = self.data.take() {
            // 数据已经就绪
            Poll::Ready(data)
        } else {
            // 没有数据，注册 Waker
            // wake() 被调用时，执行器会再次 poll 这个 Future
            let waker = cx.waker().clone();
            
            // 模拟异步 I/O 完成后调用 wake()
            std::thread::spawn(move || {
                std::thread::sleep(std::time::Duration::from_millis(50));
                println!("[IoFuture] I/O 完成，调用 wake()");
                waker.wake();
            });
            
            Poll::Pending
        }
    }
}

fn main() {
    let rt = tokio::runtime::Builder::new_current_thread()
        .build()
        .unwrap();
    
    rt.block_on(async {
        let io = IoFuture { data: Some("I/O 数据已就绪".to_string()) };
        let result = io.await;
        println!("结果：{}", result);
    });
}

use std::future::Future;
use std::task::{Context, Poll, RawWaker, RawWakerVTable, Waker};
use std::pin::Pin;
use std::sync::atomic::{AtomicBool, Ordering};

// ============== 简化的 Waker 实现 ==============

static READY: AtomicBool = AtomicBool::new(false);

fn signal_ready() {
    READY.store(true, Ordering::SeqCst);
}

unsafe fn clone_waker(_: *const ()) -> RawWaker {
    RawWaker::new(std::ptr::null(), &VTABLE)
}

unsafe fn wake_waker(_: *const ()) {
    signal_ready();
}

unsafe fn wake_by_ref_waker(_: *const ()) {
    signal_ready();
}

unsafe fn drop_waker(_: *const ()) {}

static VTABLE: RawWakerVTable = RawWakerVTable::new(
    clone_waker,
    wake_waker,
    wake_by_ref_waker,
    drop_waker,
);

fn waker() -> Waker {
    unsafe { Waker::from_raw(RawWaker::new(std::ptr::null(), &VTABLE)) }
}

// ============== 简化的执行器 ==============

fn block_on<F: Future>(mut future: F) -> F::Output {
    // 每次 poll 前重置信号
    READY.store(false, Ordering::SeqCst);
    
    // 将 future Pin 到栈上
    let mut future = Pin::new(&mut future);
    
    // 构造 Context（需要有效的 Waker）
    let waker = waker();
    let mut cx = Context::from_waker(&waker);
    
    loop {
        match future.as_mut().poll(&mut cx) {
            Poll::Ready(output) => {
                println!("[执行器] Future 完成！");
                return output;
            }
            Poll::Pending => {
                // 检查信号是否被触发
                if READY.load(Ordering::SeqCst) {
                    println!("[执行器] 收到唤醒信号，再次 poll");
                    READY.store(false, Ordering::SeqCst);
                    // 继续循环，再次 poll
                } else {
                    // 没有信号，让出 CPU（简单处理）
                    std::hint::spin_loop();
                }
            }
        }
    }
}

fn main() {
    println!("=== 简化的 Future 执行器演示 ===\n");
    
    block_on(async {
        println!("[Task] 创建异步任务");
        
        // 使用 tokio 的 sleep 来测试 Waker
        tokio::time::sleep(std::time::Duration::from_millis(50)).await;
        println!("[Task] sleep 完成");
        
        tokio::time::sleep(std::time::Duration::from_millis(50)).await;
        println!("[Task] 第二个 sleep 完成");
        
        "全部完成！"
    });
}

执行器的局限性

这个简化的执行器有几个问题：

  1. 忙等待：用 spin_loop 等待 Waker 触发，浪费 CPU
  2. 单线程：无法并行执行多个任务
  3. 简化版 Waker：不支持多个任务唤醒不同的 Future

真实的执行器（如 Tokio）使用：

  • 操作系统 I/O 多路复用（epoll/kqueue/IOCP）
  • 线程池
  • 精确的任务队列和唤醒机制

use std::future::Future;
use std::task::{Context, Poll, Waker, RawWaker, RawWakerVTable};
use std::pin::Pin;
use std::sync::{Arc, atomic::{AtomicBool, Ordering}};
use std::thread;
use std::collections::VecDeque;
use std::sync::Mutex;

type Task = Pin<Box<dyn Future<Output = ()> + Send>>;

struct SimpleExecutor {
    tasks: Arc<Mutex<VecDeque<Task>>>,
    ready_flag: Arc<AtomicBool>,
}

impl SimpleExecutor {
    fn new() -> Self {
        SimpleExecutor {
            tasks: Arc::new(Mutex::new(VecDeque::new())),
            ready_flag: Arc::new(AtomicBool::new(false)),
        }
    }

    fn spawn(&self, future: impl Future<Output = ()> + Send + 'static) {
        let task = Box::pin(future);
        self.tasks.lock().unwrap().push_back(task);
        self.ready_flag.store(true, Ordering::SeqCst);
    }

    fn run(&self) {
        let waker = self.make_waker();
        let mut cx = Context::from_waker(&waker);

        loop {
            let task = {
                let mut tasks = self.tasks.lock().unwrap();
                tasks.pop_front()
            };

            match task {
                Some(mut future) => {
                    if future.as_mut().poll(&mut cx).is_pending() {
                        // 如果 pending，重新放回队列
                        let mut tasks = self.tasks.lock().unwrap();
                        tasks.push_back(future);
                        // 让出一下 CPU
                        thread::yield_now();
                    }
                }
                None => {
                    // 没有任务了
                    if !self.ready_flag.load(Ordering::SeqCst) {
                        // 真的没任务了，退出
                        break;
                    }
                    self.ready_flag.store(false, Ordering::SeqCst);
                    thread::yield_now();
                }
            }
        }
    }

    fn make_waker(&self) -> Waker {
        let flag = self.ready_flag.clone();
        
        unsafe fn clone(_: *const ()) -> RawWaker {
            RawWaker::new(std::ptr::null(), &VTABLE)
        }
        unsafe fn wake(_: *const ()) {
            flag.store(true, Ordering::SeqCst);
        }
        unsafe fn wake_by_ref(_: *const ()) {
            flag.store(true, Ordering::SeqCst);
        }
        unsafe fn drop(_: *const ()) {}

        static VTABLE: RawWakerVTable = RawWakerVTable::new(clone, wake, wake_by_ref, drop);
        
        unsafe { Waker::from_raw(RawWaker::new(std::ptr::null(), &VTABLE)) }
    }
}

fn main() {
    println!("=== 多任务执行器演示 ===\n");
    
    let executor = SimpleExecutor::new();
    
    executor.spawn(async {
        println!("任务 A 开始");
        tokio::time::sleep(std::time::Duration::from_millis(20)).await;
        println!("任务 A 完成");
    });
    
    executor.spawn(async {
        println!("任务 B 开始");
        tokio::time::sleep(std::time::Duration::from_millis(30)).await;
        println!("任务 B 完成");
    });
    
    executor.spawn(async {
        println!("任务 C 开始");
        tokio::time::sleep(std::time::Duration::from_millis(10)).await;
        println!("任务 C 完成");
    });
    
    executor.run();
    
    println!("\n所有任务执行完毕！");
}

impl Future for CounterFuture {
    type Output = u32;

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        self.count += 1;
        
        if self.count >= self.max {
            Poll::Ready(self.count)

            // 模拟计数器"自然增长"，通过 wake_by_ref 继续
            let waker = cx.waker().clone();
            std::thread::spawn(move || {
                std::thread::sleep(std::time::Duration::from_millis(1));
                waker.wake_by_ref();
            });
            Poll::Pending
        }
    }
}

struct JoinFuture<A, B> {

    fut_b: Option<B>,
    ready_a: Arc<AtomicBool>,
    ready_b: Arc<AtomicBool>,
    waker_a: Option<Waker>,
    waker_b: Option<Waker>,

impl<A, B, Ta, Tb> Future for JoinFuture<A, B> 
where
    A: Future<Output = Ta>,
    B: Future<Output = Tb>,
{
    type Output = (Ta, Tb);

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        // 实现逻辑：
        // 1. poll fut_a；如果完成，标记 ready_a
        // 2. poll fut_b；如果完成，标记 ready_b
        // 3. 如果两者都完成，返回 Ready((a, b))
        unimplemented!()
    }
}

fn join<A, B>(a: A, b: B) -> JoinFuture<A, B> 
where
    A: Future + Unpin,
    B: Future + Unpin,
{
    JoinFuture {
        fut_a: Some(a),
        fut_b: Some(b),
        ready_a: Arc::new(AtomicBool::new(false)),
        ready_b: Arc::new(AtomicBool::new(false)),
        waker_a: None,
        waker_b: None,
    }
}

        async { 2 },

impl<A, B, Ta, Tb> Future for JoinFuture<A, B>
where
    A: Future<Output = Ta>,
    B: Future<Output = Tb>,
{
    type Output = (Ta, Tb);

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output> {
        // 克隆 waker 用于后续唤醒
        let waker_a = cx.waker().clone();
        let waker_b = cx.waker().clone();

        // Poll fut_a
        if let Some(ref mut fut_a) = self.fut_a {
            let pinned_a = Pin::new(fut_a);
            if let Poll::Ready(val_a) = pinned_a.poll(&mut Context::from_waker(&waker_a)) {
                self.ready_a.store(true, Ordering::SeqCst);
                self.fut_a = None;  // 取走 fut_a
                
                // 等待 fut_b
                if let Some(ref mut fut_b) = self.fut_b {
                    let pinned_b = Pin::new(fut_b);
                    if let Poll::Ready(val_b) = pinned_b.poll(&mut Context::from_waker(&waker_b)) {
                        return Poll::Ready((val_a, val_b));
                    }
                }
                // fut_b 还 pending，但 fut_a 完成了，无法继续
                // 这里简化处理，实际应用中应该更精细
            }
        }

        // Poll fut_b
        if let Some(ref mut fut_b) = self.fut_b {
            let pinned_b = Pin::new(fut_b);
            if let Poll::Ready(val_b) = pinned_b.poll(&mut Context::from_waker(&waker_b)) {
                self.ready_b.store(true, Ordering::SeqCst);
                self.fut_b = None;
                
                if self.fut_a.is_none() {
                    // fut_a 已经完成，但结果在上面的分支里丢失了
                    // 这个简化实现有缺陷，需要更完整的 Join 实现
                }
            }
        }

        Poll::Pending
    }
}

        // 注意：返回的 Future 生命周期与 self 关联
        async move {
            let data = self.data;  // 移动 data 到 async 块中
            println!("future 中的 data: {}", data);

// 思考题：执行器如何调度多个任务？

// 场景：有 1000 个 Future，每个 Future 在 poll 时返回 Pending
// 执行器应该：

// A. 对每个 Future 连续调用 poll，直到它完成
// B. 对每个 Future 调用一次 poll，然后换下一个
// C. 使用 I/O 多路复用等待某个 Future 就绪后再 poll
// D. B + C 结合

// 选择正确的实现方式，并解释原因
参考答案

答案是 D（B + C 结合）。

在单线程执行器中，采用协作式调度：每轮对每个任务调用一次 poll。

  • 每次 poll 后，如果任务仍为 Pending，将其放回队列尾部
  • 这样可以保证所有任务都有机会被处理

在多线程执行器中，结合 I/O 多路复用：

  • 通过 epoll/kqueue/IOCP 监听 I/O 事件
  • 当某个 fd 可读/可写时，唤醒对应任务
  • 大幅减少忙等待，提高效率

这就是 Tokio 等生产级执行器的工作原理。

今日感悟

1. Future 的 poll 模型是理解异步的关键

今天最大的收获是理解了 poll 模型：Future 不需要阻塞等待，它在数据未就绪时返回 Pending，然后通过 Waker 在就绪时被唤醒再次 poll。这种"拉取"（pull）模型与传统的"推送"（push）模型形成对比。

2. Pin 是为了安全处理自引用

Pin 看起来复杂，但它要解决的问题很清晰：如果一个 Future 包含指向自己内部数据的引用，而 Future 可能在 .await 期间被暂停，那么这个 Future 不能在内存中移动。Pin 就是为了解决这个问题。

3. 执行器是连接 Future 和操作系统的桥梁

执行器的职责很清晰：管理任务队列、响应 Waker 事件、调用 poll。它把操作系统的 I/O 多路复用（epoll/kqueue）和 Rust 的 Future 模型连接起来，让开发者可以用声明式的方式编写异步代码。

4. 手动实现 Future 有助于深入理解

通过实现 CounterFuture 和 SimpleExecutor，我理解了标准库 Future trait 的设计动机。实际的 Tokio 等库做了大量优化，但核心机制与我们的简化版本相同。

5. async 代码的生命周期需要特别注意

async 块会捕获环境中的引用，这些引用的生命周期必须与 async 块本身一致。这是一个容易出错的点——如果闭包捕获了局部的 &str，返回的 Future 不能持有这个引用超过闭包的生命周期。
