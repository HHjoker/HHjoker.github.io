回顾：执行器与 Tokio

昨天我们手写了一个简化执行器，理解了 Future 的 poll 机制、Waker 唤醒和 Pin 的作用。

真实世界的执行器要比我们写的复杂得多。Tokio 是 Rust 生态中最成熟的生产级异步运行时。今天我们就深入了解 Tokio 的架构。

一、Tokio 简介

什么是 Tokio？

Tokio 是一个异步运行时，提供：

  • 多线程任务调度器
  • 异步 I/O（TCP/UDP/Unix socket）
  • 定时器（sleep、timeout）
  • 文件系统操作（异步）
  • 同步原语（Mutex、channel 等）

Tokio vs 其他选择

运行时 特点
Tokio 最流行、生态丰富、生产级稳定
async-std 设计风格接近标准库，概念清晰
smol 轻量级、易于理解
glommio Linux io_uring 支持，高性能

Tokio 是目前最主流的选择，大多数生产项目都用它。

二、Tokio 运行模式

Tokio 有两种运行模式：

1. 单线程模式（current_thread）

只在一个线程上运行 Future，适合简单场景：

use tokio::runtime::Builder;

fn main() {
    // 创建单线程运行时
    let runtime = Builder::new_current_thread()
        .build()
        .unwrap();

    // 在这个线程上运行 Future
    runtime.block_on(async {
        println!("在单线程运行时中运行");
        tokio::time::sleep(std::time::Duration::from_millis(10)).await;
        println!("完成！");
    });
}

use tokio;

#[tokio::main]
async fn main() {
    // spawn 返回 JoinHandle，可以等待任务完成
    let handle = tokio::spawn(async {
        // 这个 async 块会在某个工作线程上执行
        tokio::time::sleep(std::time::Duration::from_millis(100)).await;
        42
    });

    // 在主任务中做其他事
    println!("等待 spawn 的任务...");

    // 等待并获取结果
    let result = handle.await.unwrap();
    println!("spawn 任务返回: {}", result);
}

    let _ = handle.await;

}

#[tokio::main]
async fn main() {
    let mut handles = vec![];

    for i in 0..5 {
        let handle = tokio::spawn(async move {
            tokio::time::sleep(std::time::Duration::from_millis(10 * i as u64)).await;
            format!("任务 {} 完成", i)
        });
        handles.push(handle);
    }

    // 等待所有任务
    for handle in handles {
        let result = handle.await.unwrap();
        println!("{}", result);
    }
}

                    ┌─────────────────────────────────────┐
                    │            Tokio Runtime             │
                    │                                      │
  spawn(Future) ──→ │   ┌──────────────────────────────┐  │
                    │   │        Task Queue             │  │
                    │   │  (每个 Worker Thread 一个)     │  │
                    │   └──────────────────────────────┘  │
                    │              ↓  ↓  ↓                │
                    │   ┌────────┐ ┌────────┐ ┌────────┐  │
                    │   │Worker 1│ │Worker 2│ │Worker 3│  │
                    │   └────────┘ └────────┘ └────────┘  │
                    │   (poll Future)  (poll Future)      │
                    └─────────────────────────────────────┘

每个 Worker Thread：
1. 从自己的本地队列取任务
2. poll 任务（调用 Future.poll）
3. 如果 Future 返回 Pending，任务留在队列
4. 如果 Future 完成，任务从队列移除

steal（任务窃取）

如果某个工作线程的队列空了，它会从其他线程的队列"窃取"任务：

1
2
3
4
5
6
7
8

// Tokio 的调度策略：
// 1. 优先从本地队列取任务
// 2. 本地队列空时，从其他线程窃取
// 3. 窃取时随机选择目标队列

// 这保证了：
// - 任务尽量在创建它的线程上执行（缓存友好）
// - 负载均衡（没有线程空闲而其他线程忙）

五、spawn_blocking：处理阻塞代码

为什么要用 spawn_blocking？

有些操作是阻塞的（CPU密集计算、调用同步库等），不应该在异步线程上执行：

⚠️ 问题场景：
┌────────────────────────────────────────┐
│  假设你在异步线程中执行 CPU 密集计算：   │
│                                        │
│  tokio::spawn(async {                  │
│      let result = cpu_heavy_task();    │ ← 这个线程被阻塞
│      result                             │   其他任务无法使用
│  });                                    │   这个线程
└────────────────────────────────────────┘

解决方案：用 spawn_blocking 移到专用线程：
┌────────────────────────────────────────┐
│  tokio::spawn_blocking(|| {            │
│      // 在专用线程执行，不阻塞异步线程   │
│      cpu_heavy_task()                  │
│  });                                   │
└────────────────────────────────────────┘

use tokio;

#[tokio::main]
async fn main() {
    // 使用 spawn_blocking 处理阻塞操作
    let result = tokio::task::spawn_blocking(|| {
        // 这个闭包会在一个专门的"阻塞线程"上执行
        println!("在阻塞线程中执行...");
        
        // 模拟 CPU 密集计算
        let sum: u64 = (1..=1000000).sum();
        
        std::thread::sleep(std::time::Duration::from_millis(100));
        
        sum
    }).await.unwrap();

    println!("spawn_blocking 结果: {}", result);
}

spawn vs spawn_blocking 对比

场景 使用 spawn 使用 spawn_blocking
异步 I/O（HTTP请求、数据库查询） ✅ ❌
CPU 密集计算 ❌ ✅
调用同步阻塞库 ❌ ✅
并行处理独立任务 ✅ ❌

use tokio;
use tokio::fs;

#[tokio::main]
async fn main() -> std::io::Result<()> {
    // Tokio 的 fs 模块是异步的，但底层也是通过 spawn_blocking 实现
    let contents = fs::read_to_string("data.txt").await?;
    
    println!("文件内容: {}", contents);
    
    // 对于完全同步的代码，自己用 spawn_blocking
    let hash = tokio::task::spawn_blocking(|| {
        // 假设有个同步的哈希计算
        compute_hash_sync()
    }).await.unwrap();
    
    println!("哈希值: {}", hash);
    
    Ok(())
}

fn compute_hash_sync() -> String {
    // 模拟同步计算
    std::thread::sleep(std::time::Duration::from_millis(50));
    "abc123".to_string()
}

        .thread_name("outer")
        .build()
        .unwrap();

    outer_runtime.block_on(async {
        println!("在外层运行时中");

        // ⚠️ 在异步上下文中创建新运行时需要谨慎

        let inner_runtime = Builder::new_current_thread()
            .thread_name("inner")
            .build()
            .unwrap();

        inner_runtime.block_on(async {
            println!("在内层 current_thread 运行时中");
        });

        println!("回到外层运行时");
    });
}

use tokio;

#[tokio::main]
async fn main() {
    let handle = tokio::spawn(async {
        loop {
            println!("任务运行中...");
            tokio::time::sleep(std::time::Duration::from_millis(100)).await;
        }
    });

    // 3 秒后取消任务
    tokio::time::sleep(std::time::Duration::from_secs(3)).await;
    
    println!("开始取消任务...");
    handle.abort();
    
    // join 会返回错误，因为任务被取消了
    let result = handle.await;
    
    match result {
        Ok(_) => println!("任务正常完成"),

use tokio::{spawn, time::{sleep, Duration}};

#[tokio::main]
async fn main() {
    let long_task = spawn(async {
        loop {
            println!("长时间任务运行中");
            sleep(Duration::from_millis(100)).await;
        }
    });

    let timeout_task = spawn(async {
        sleep(Duration::from_secs(2)).await;
        println!("timeout 任务完成");
    });

    // 等待任意一个完成
    let mut handles = vec![long_task, timeout_task];
    
    while !handles.is_empty() {
        for i in 0..handles.len() {
            match handles[i].now_or_never() {
                Some(Ok(())) => {
                    println!("任务 {} 完成", i);
                }
                Some(Err(e)) => {
                    println!("任务 {} 出错: {:?}", i, e);
                }
                None => {}
            }
        }
        
        if handles.len() == 2 {
            // 两个都还没完成，继续等
            sleep(Duration::from_millis(10)).await;
        }
    }
}

今日感悟

1. Tokio 把复杂的事情隐藏了

#[tokio::main] 一行背后是：多线程运行时构建、任务队列管理、Waker 注册、I/O 多路复用……但我们只需要写 async fn，Tokio 替我们搞定一切。

2. spawn 与 spawn_blocking 的选择是性能关键

在生产代码中，如果把 CPU 密集任务用 spawn 放到异步线程池，会导致整个线程池被占住，其他任务无法推进。spawn_blocking 把这类任务隔离到专用线程，保持异步线程池的高效。

3. Tokio 的调度比我想的更智能

通过任务窃取（work stealing），Tokio 保证即使某些任务运行时间很长，其他线程也不会闲着。这比自己手写的简单轮询调度要复杂得多，但 Tokio 已经帮你做好了。

4. 理解运行时的嵌套边界很重要

在已有 Tokio 运行时中，尽量不要再创建 multi-thread 运行时，这会导致线程爆炸。但创建 current_thread 运行时是可以的，用于隔离某些任务。

5. 超时和取消是异步编程的必备技能

timeout 和 AbortHandle 是控制异步任务生命周期的重要工具，生产代码中几乎总是需要处理超时场景，防止任务永远挂起。
