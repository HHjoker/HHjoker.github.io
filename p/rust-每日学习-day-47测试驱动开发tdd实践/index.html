什么是 TDD？

测试驱动开发（Test-Driven Development） 是一种软件开发方法论，核心思想是：先写测试，再写代码。

传统开发 vs TDD

传统开发流程：

1
2
3

写代码 → 手动测试 → 发现问题 → 修复 → 再测试 → 完成
         ↓
    测试是事后验证

TDD 流程：

1
2
3

写测试（失败）→ 写代码（通过）→ 重构 → 下一个测试 → 完成
     ↓
  测试驱动设计

为什么使用 TDD？

优势

  1. 更好的设计 - 从使用者角度思考 API
  2. 更高的覆盖率 - 每个功能都有测试
  3. 更安全重构 - 测试保证不破坏现有功能
  4. 文档即测试 - 测试展示如何使用代码
  5. 减少调试时间 - 小步快跑，问题容易定位

挑战

  1. 思维转变 - 需要习惯先写测试
  2. 初期慢 - 前期投入时间多
  3. 不适合所有场景 - UI、原型开发可能不适合
  4. 需要自律 - 坚持红 - 绿 - 重构循环

pub fn calculate(expression: &str) -> Result<i32, &'static str> {
    let parts: Vec<&str> = expression.split_whitespace().collect();
    
    if parts.len() != 3 {
        return Err("Invalid expression format");
    }
    
    let left: i32 = parts[0].parse().map_err(|_| "Invalid left operand")?;
    let operator = parts[1];
    let right: i32 = parts[2].parse().map_err(|_| "Invalid right operand")?;
    
    match operator {
        "+" => Ok(left + right),
        "-" => Ok(left - right),
        _ => Err("Unsupported operator"),
    }
}

#[derive(Debug)]
struct Expression {
    left: i32,
    operator: char,
    right: i32,
}

fn parse_expression(expr: &str) -> Result<Expression, &'static str> {
    let parts: Vec<&str> = expr.split_whitespace().collect();
    
    if parts.len() != 3 {
        return Err("Invalid expression format");
    }
    
    let left = parts[0].parse().map_err(|_| "Invalid left operand")?;
    let operator = parts[1].chars().next().ok_or("Invalid operator")?;
    let right = parts[2].parse().map_err(|_| "Invalid right operand")?;
    
    Ok(Expression { left, operator, right })
}

pub fn calculate(expression: &str) -> Result<i32, &'static str> {
    let expr = parse_expression(expression)?;
    
    match expr.operator {
        '+' => Ok(expr.left + expr.right),
        '-' => Ok(expr.left - expr.right),
        _ => Err("Unsupported operator"),
    }
}

pub fn calculate(expression: &str) -> Result<i32, &'static str> {
    let expr = parse_expression(expression)?;
    
    match expr.operator {
        '+' => Ok(expr.left + expr.right),
        '-' => Ok(expr.left - expr.right),
        '*' => Ok(expr.left * expr.right),
        '/' => {
            if expr.right == 0 {
                Err("Division by zero")
            } else {
                Ok(expr.left / expr.right)
            }
        }
        _ => Err("Unsupported operator"),
    }
}

impl TodoList {
    pub fn add(&mut self, title: &str) -> Todo {
        let todo = Todo {
            id: self.next_id,
            title: title.to_string(),
            status: Status::Pending,

    }

impl TodoList {
    pub fn complete(&mut self, id: u32) -> Result<(), &'static str> {
        match self.items.iter_mut().find(|t| t.id == id) {
            Some(todo) => {
                todo.status = Status::Completed;
                Ok(())
            }
            None => Err("Todo not found"),
        }
    }

    pub fn get(&self, id: u32) -> Option<&Todo> {
        self.items.iter().find(|t| t.id == id)
    }
}

impl TodoList {

        let pos = self.items.iter().position(|t| t.id == id);

impl std::fmt::Display for TodoList {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        if self.items.is_empty() {
            writeln!(f, "No todos yet!")?;
        } else {
            for todo in &self.items {
                let status = match todo.status {
                    Status::Pending => "⬜",
                    Status::Completed => "✅",
                };
                writeln!(f, "{} [{}] {}", status, todo.id, todo.title)?;
            }
        }
        Ok(())
    }
}

3. 遵循 FIRST 原则

  • Fast - 测试要快
  • Independent - 测试独立
  • Repeatable - 可重复运行
  • Self-validating - 自动验证
  • Timely - 及时编写

关键概念理解

1. TDD 不是关于测试

TDD 的核心是设计，不是测试。测试是副产品，设计才是目标。

2. 红 - 绿 - 重构的节奏 

1
2
3

红（5 分钟） → 绿（10 分钟） → 重构（15 分钟）
     ↓              ↓              ↓
  写测试        写代码        优化代码

保持小步快跑，每个循环不超过 30 分钟。

3. 测试金字塔 

1
2
3
4
5
6
7

        /\
       /  \      E2E 测试（少量）
      /----\
     /      \    集成测试（适量）
    /--------\
   /          \   单元测试（大量）
  /------------\
  • 底层：大量快速的单元测试
  • 中层：适量的集成测试
  • 顶层：少量的端到端测试

4. TDD 的局限性

不适合 TDD 的场景：

  • UI/UX 探索性开发
  • 快速原型
  • 算法研究
  • 一次性脚本

今日感悟

今天的 TDD 实践让我深刻体会到：

  1. 测试驱动的是设计，不是验证 - 当我先写测试时，我被迫从使用者角度思考 API。这自然产生了更好的设计。

  2. 小步快跑的力量 - 每次只写一个失败的测试，然后让它通过。这种节奏让我始终保持专注，不会陷入"写了一堆代码才发现方向错了"的困境。

  3. 重构的底气 - 有了完整的测试覆盖，重构不再是冒险。我可以放心地优化代码结构，因为测试会告诉我是否破坏了功能。

  4. 文档的另一种形式 - 测试本身就是最好的文档。新开发者可以通过测试快速理解代码应该如何工作。

  5. 需要自律 - TDD 在初期确实感觉"慢"。但长远来看，它节省了大量调试和修复 bug 的时间。关键在于坚持。

  6. 不是银弹 - TDD 不是万能的。对于探索性工作、UI 设计等场景，可能需要更灵活的方法。知道何时使用 TDD 同样重要。

TDD 是一种需要练习的技能。今天的练习只是开始，真正的掌握需要在实际项目中持续应用。

明天进入性能测试，从另一个维度保证代码质量。期待！
